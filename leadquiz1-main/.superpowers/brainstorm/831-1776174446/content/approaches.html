Choose a Prototype Approach

All three produce a click-through HTML prototype. They differ in structure and fidelity level.

A

Single-File State Machine Recommended

One HTML file. JavaScript drives screen transitions with smooth fade/slide animations. Fully self-contained — open in any browser, send as one file, no server needed.

Pros

  • Zero dependencies, one file to share
  • Smooth animated transitions between screens
  • Can simulate real interaction (MCQ selection, progress bar)
  • Closest to the real H5 feel

Cons

  • Larger single file
B

Multi-Page HTML Folder

Each screen is its own HTML file linked together. Simple navigation, easy to edit or hand off individual screens separately.

Pros

  • Each screen is isolated, easy to edit
  • Good for design handoff (share individual screens)

Cons

  • Requires a folder, not a single file
  • No shared state (progress bar, selections don't carry over)
  • Less immersive — full page reloads between screens
C

Scrollable Preview Doc

All screens stacked vertically in one long HTML document. No JS transitions — just visual design reference. Think: "Figma export".

Pros

  • Simplest output, easy to screenshot
  • Good for design review / stakeholder sign-off

Cons

  • Not an interactive flow — you said click-through
  • Less useful for user testing or sharing
